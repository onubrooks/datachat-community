# FND-008: Query DataPoint Generation V1

**Initiative ID:** FND-008  
**Status:** In Progress  
**Phase:** Phase A follow-on (answer-quality hardening)  
**Last Updated:** 2026-04-04

---

## 1. Problem

DataChat is strong when retrieval finds a high-signal datapoint or when the schema is simple enough for ad hoc SQL generation. It is weaker on reusable business questions that follow common analytics shapes, such as:

- top-N by a business dimension;
- weekly or monthly trend over a recent window;
- net-flow style calculations from paired inflow/outflow columns;
- grouped aggregations that should be answered with a stable query pattern instead of regenerated from scratch.

Today, profiling primarily emits schema and business datapoints. That leaves a gap between table understanding and reusable answer plans.

## 2. Scope

In scope:

- Generate `Query` datapoints from profiled schemas using deterministic heuristics.
- Cover first-pass reusable analytics patterns:
  - top-N by primary dimension and measure;
  - weekly trend by primary time column and measure;
  - net-flow grouped trend when paired inflow/outflow measures exist.
- Attach parameter metadata so query templates can absorb values like `top 5` and `last 8 weeks` from the user question.
- Persist generated query datapoints through profiling API, CLI, and tool flows.
- Add narrow QA eval coverage aligned to question-bank prompts.

Out of scope:

- free-form LLM-only query generation packs;
- broad domain-specific pack generation beyond compact grocery/fintech-friendly heuristics;
- training UI redesign for authoring arbitrary new datapoint classes;
- major retrieval/ranking redesign outside what is required to consume generated query datapoints.

## 3. Proposed Design

### 3.1 Generation lane

Extend profiling generation output to include `query_datapoints` alongside schema and business datapoints.

Generation should be deterministic-first:

1. detect candidate time column;
2. detect candidate business dimension column;
3. detect candidate numeric measure columns;
4. emit compact query-template families from those candidates.

### 3.2 First-pass query families

#### A. Top-N by dimension

Example shape:

- `Top customer segments by deposits`
- `Top stores by revenue`

Template contract:

- grouped aggregation by one dimension;
- sorted descending on one measure;
- parameterized `top_n` with default;
- tags like `top_n`, `ranking`, `aggregation`.

#### B. Weekly trend

Example shape:

- `Weekly deposit trend`
- `Weekly revenue trend`

Template contract:

- recent rolling window;
- grouped by week on the chosen time column;
- parameterized `lookback_weeks` with default;
- connector-aware weekly syntax via backend variants where needed.

#### C. Net flow by dimension and week

Example shape:

- `Deposits, withdrawals, and net flow by customer segment for the last 8 weeks`

Eligibility:

- paired inflow/outflow numeric columns detected from names like `deposit`, `withdraw`, `credit`, `debit`, `inflow`, `outflow`.
- requires one time column and one grouping dimension.

Template contract:

- aggregate inflow and outflow separately;
- compute derived `net_flow`;
- grouped by week and dimension;
- parameterized `lookback_weeks`.

### 3.3 Parameter filling

Use deterministic parsing in the SQL agent to map user phrasing into template parameters.

Examples:

- `top 5`, `first 10` -> `top_n`
- `last 8 weeks`, `6 months`, `30 days` -> window parameters

If no value is present in the prompt, keep the template default.

### 3.4 Naming and metadata

Each generated query datapoint must include:

- stable generated ID;
- readable business name;
- related tables;
- tags and synonyms oriented toward likely question phrasing;
- metadata for:
  - `query_family`
  - `primary_dimension`
  - `primary_measure`
  - `time_grain`
  - `generated_from_profile`
  - `parameters`

## 4. API and Contract Changes

- Extend `GeneratedDataPoints` with `query_datapoints`.
- Ensure profiling and generation APIs persist `Query` datapoints the same way they persist schema/business datapoints.
- Ensure generation summaries and quality tools count query datapoints.
- Ensure SQL agent template execution path can use generated parameter metadata.

## 5. Data Model and Storage Changes

No new top-level storage system is required.

Changes:

- generated output model now includes `query_datapoints`;
- profiling persistence paths must include `Query` records;
- dedupe logic must consider query datapoint IDs separately from schema/business artifacts.

## 6. Safety and Quality Controls

- Keep generated query families compact to avoid retrieval noise.
- Prefer deterministic templates over free-form LLM query synthesis when generating query datapoints.
- Do not emit net-flow templates unless paired measures are strongly indicated.
- Keep expected SQL assertions broad in evals so connector differences do not cause false failures.

## 7. Acceptance Criteria

Functional:

- [ ] Profiling can generate `Query` datapoints without manual authoring.
- [ ] Generated query datapoints persist through API, CLI, and tool-based generation flows.
- [ ] SQL agent can fill template defaults from the user question for at least `top_n` and rolling-window parameters.

Quality:

- [ ] At least one top-N style prompt is answered through a generated query datapoint path.
- [ ] At least one weekly-trend style prompt is answered through a generated query datapoint path.
- [ ] At least one net-flow style prompt is answered through a generated query datapoint path when schema supports it.

## 8. Test Plan

Unit:

- generator emits top-N, weekly-trend, and net-flow query datapoints when schema signals exist;
- SQL template parameter filling extracts values from prompt text;
- dedupe logic keeps query datapoints isolated from schema/business dedupe.

Integration:

- profiling route persists generated query datapoints;
- CLI generation path persists generated query datapoints;
- quality summary reflects query datapoint counts.

Eval/manual:

- add fintech QA eval prompts aligned to grouped, ranked, and temporal patterns;
- validate generated artifacts and resulting SQL on at least one profiled dataset.

## 9. Rollout and Rollback

Rollout:

1. Enable generation in private and community behind the existing profiling path.
2. Validate with targeted unit/integration tests.
3. Run manual profiling against fintech and grocery-style schemas.
4. Expand query families only after retrieval noise stays acceptable.

Rollback:

- disable query generation by returning only schema/business datapoints from profiling generator;
- keep profiling and retrieval codepaths otherwise unchanged.

## 10. Open Questions

- Should future versions emit monthly and daily variants explicitly, or should weekly remain the only automatic temporal family for v1?
- Should generated query datapoints receive lower retrieval priority than reviewed/manual query datapoints?
- What is the right ceiling for generated query datapoints per table before recall starts hurting precision?

## 11. Implementation Phases

Phase 1 (High priority): Core generation model

- extend generated output model;
- add deterministic heuristics for time, dimension, and measure selection;
- emit top-N, weekly-trend, and net-flow query datapoints.

Phase 2 (High priority): Runtime integration

- persist generated query datapoints through API, CLI, and tool flows;
- include them in summaries and dedupe;
- fill query-template parameters from user phrasing.

Phase 3 (Medium priority): Coverage and evals

- add QA eval pack for generated query patterns;
- validate question-bank prompts against generated query artifacts;
- tighten naming and metadata based on observed misses.
